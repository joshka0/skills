# Planning Workflow

Use this skill for the product/issue lifecycle:

```text
conversation or spec → PRD
PRD or plan → vertical-slice issues
incoming issue → triage state
ready issue → agent brief
rejected enhancement → out-of-scope memory
```

## Setup dependency

The issue tracker and triage label vocabulary should have been provided to you. If not, run `/setup-matt-pocock-skills` before publishing, labeling, closing, or writing public issue-tracker comments.

## Modes

- `prd` — synthesize current context into a PRD.
- `issues` — break a plan, PRD, or spec into independently grabbable vertical slices.
- `triage` — move incoming issues through the triage state machine.
- `agent-brief` — write a durable brief for an AFK agent.
- `out-of-scope` — record rejected enhancement decisions for future triage.

## Hard rules

- Use current context; do not run broad interviews.
- Ask only for missing decisions that materially change the PRD, issue breakdown, triage state, or public issue-tracker output.
- Use the project domain glossary and respect ADRs when exploring a codebase.
- Avoid specific file paths or code snippets in durable PRDs/issues unless a prototype encodes a decision more precisely than prose can.
- For public comments, closing issues, or writing `.out-of-scope/`, preview exact public text unless the maintainer already supplied it.
- Every triage comment or issue comment generated during triage must start with:

```markdown
> *This was generated by AI during triage.*
```

## Output

Report the active mode, the issue-tracker action you will take or took, and any follow-up decisions needed from the maintainer.
