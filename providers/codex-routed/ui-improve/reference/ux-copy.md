# UX Copy

Rewrite unclear interface text — errors, labels, CTAs, help, and microcopy. Good UX writing is invisible: users understand immediately without noticing the words.

## Error Messages

**Bad**: `Error 403: Forbidden`
**Good**: `You don't have permission to view this page. Contact your admin for access.`

**Bad**: `Invalid input`
**Good**: `Email addresses need an @ symbol. Try: name@example.com`

Principles:
- Explain what went wrong in plain language
- Suggest how to fix it
- Don't blame the user
- Include examples when helpful
- Link to help/support if applicable

## Form Labels & Instructions

**Bad**: `DOB (MM/DD/YYYY)`
**Good**: `Date of birth` (with placeholder showing format)

**Bad**: `Enter value here`
**Good**: `Your email address`

Principles:
- Clear, specific labels (not generic placeholders)
- Show format expectations with examples
- Explain why you're asking (when not obvious)
- Put instructions before the field, not after
- Keep required field indicators clear

## Button & CTA Text

**Bad**: `Click here` | `Submit` | `OK`
**Good**: `Create account` | `Save changes` | `Got it, thanks`

Principles:
- Describe the action specifically
- Use active voice (verb + noun)
- Match user's mental model
- Be specific ("Save" is better than "OK")

## Help Text & Tooltips

**Bad**: `This is the username field`
**Good**: `Choose a username. You can change this later in Settings.`

Principles:
- Add value (don't just repeat the label)
- Answer the implicit question ("What is this?" or "Why do you need this?")
- Keep brief but complete
- Link to detailed docs if needed

## Empty States

**Bad**: `No items`
**Good**: `No projects yet. Create your first project to get started.`

Principles: Explain why it's empty, show next action clearly, make it welcoming.

## Success Messages

**Bad**: `Success`
**Good**: `Settings saved! Your changes will take effect immediately.`

Principles: Confirm what happened, explain what happens next, match emotional moment.

## Loading States

**Bad**: `Loading...` (for 30+ seconds)
**Good**: `Analyzing your data... this usually takes 30–60 seconds`

Principles: Set expectations (how long), explain what's happening, show progress when possible, offer escape hatch.

## Confirmation Dialogs

**Bad**: `Are you sure?`
**Good**: `Delete 'Project Alpha'? This can't be undone.`

Principles: State the specific action, explain consequences (especially destructive ones), use clear button labels ("Delete project" not "Yes"), don't overuse (only for risky actions).

## Navigation & Wayfinding

**Bad**: `Items` | `Things` | `Stuff`
**Good**: `Your projects` | `Team members` | `Settings`

Principles: Be specific and descriptive. Use language users understand (not internal jargon). Make hierarchy clear. Consider information scent (breadcrumbs, current location).

## Clarity Principles

Every piece of copy should be:

1. **Specific**: "Enter email" not "Enter value"
2. **Concise**: Cut unnecessary words without sacrificing clarity
3. **Active**: "Save changes" not "Changes will be saved"
4. **Human**: "Oops, something went wrong" not "System error encountered"
5. **Helpful**: Tell users what to do, not just what happened
6. **Consistent**: Use same terms throughout — don't vary for variety

## Rules

- Don't use jargon without explanation
- Don't blame users ("You made an error" → "This field is required")
- Don't be vague ("Something went wrong" without explanation)
- Don't use passive voice unnecessarily
- Don't use humor for errors — be empathetic
- Don't assume technical knowledge
- Don't vary terminology — pick one term and stick with it
- Don't use placeholders as the only labels (they disappear when users type)
